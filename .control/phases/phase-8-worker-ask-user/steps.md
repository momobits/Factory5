## Phase 8 — Worker-subprocess `askUser`

- [x] 8.1 — **[ADR 0024](../../../docs/decisions/0024-worker-subprocess-ask-user.md)** worker→brain `askUser` design. Decided:
  - **Route**: MCP server (Claude CLI's official tool-extension path; net-new infra; clean abstraction; reusable for future custom tools) vs direct-stdio JSON-RPC (smaller diff; reuses existing stream parser; bespoke; harder to extend). Consider the cost of standing up the first MCP server in the codebase against the value of a shape that scales to N future custom tools.
  - **Wall-clock budget policy**: does a worker waiting on `ask_user` pause the directive's TTL? Pause `max_steps` accounting? Three options: (a) wall-clock counts as normal — long answers eat the budget (simple, harsh); (b) wait time excluded from `max_usd` and `max_steps` (matches operator intent: "the agent is correctly stopped, not thrashing"); (c) hybrid — `max_steps` paused but `max_usd` keeps accruing as the subprocess is still pinning a CLI seat. Recommendation lean: (b), with a per-question soft deadline (default 1 hour, configurable in `config.toml [budget.askUser]`) so the worker doesn't pin a seat indefinitely.
  - **Correlation contract**: `taskId` becomes mandatory in the worker→brain `ask_user` call (today it's optional in `pending_questions`). Two workers in the same directive with similar questions must each get their own answer. Brain-side proxy enforces by always passing `taskId` from the request envelope.
  - **Brain-restart recovery**: define `tasks_inflight.status = 'waiting_for_human'`. On brain startup, scan for orphaned `waiting_for_human` rows whose worker subprocess is dead (parent_pid != current_brain_pid OR pid not found); mark those tasks `aborted` with `aborted_reason = 'brain_restart_during_human_wait'`. A late-arriving answer for that pending_question writes to the row but its associated task is already dead — log "answered after task ended" and no-op (no retry).
  - **Tool whitelist**: which agent roles get `AskUser`? Recommendation: `scaffolder` / `builder` / `fixer` / `investigator` (the four that own actual writes or runtime decisions). Excluded: `triage` (no tools today), `architect` / `planner` / `reviewer` / `verifier` (already brain-checkpointed via existing `escalateBlocked`).
  - **Out of scope for ADR 0024**: replacing brain-level `askUser` (Phase 4 stays as the canonical primitive — Phase 8 is additive); changing channel answer collection (already polyglot across Discord / Telegram / CLI).
  - Output: `docs/decisions/0024-worker-subprocess-ask-user.md` + INDEX row.

- [x] 8.2 — Brain RPC endpoint. Extended existing daemon Fastify server (per ADR 0012 brain runs in `factoryd` process; daemon already imports from `@factory5/brain`, so a new daemon route is the lowest-coupling shape rather than spinning up a second server). `POST /worker/ask-user`:
  - Request schema `workerAskUserRequestSchema` + response schema `workerAskUserResponseSchema` in `packages/ipc/src/schemas.ts` (taskId mandatory per ADR 0024 §3; deadlineSeconds optional, daemon defaults 3600s). 9 schema tests.
  - Route handler in `packages/daemon/src/server.ts` with bearer-token gate (constant-time compare), `503 WORKER_ASK_USER_DISABLED` when handler not configured, `401 WORKER_AUTH_REQUIRED` before schema parse so unauthenticated callers can't probe schema. Bearer check uses pure XOR fold on equal-length strings; length-mismatch shortcut OK because token length is a public per-startup constant. 9 route tests (happy path, timed-out path, 503 disabled, 401 missing/wrong, 200 correct, 400 missing taskId, 401-before-schema, 404 IpcRequestError pass-through).
  - `buildWorkerAskUserHandler` in `packages/daemon/src/index.ts` validates `(taskId, directiveId)` lives in `tasks_inflight` (defense-in-depth), computes `deadlineAt = now + deadlineSeconds`, calls existing `askUser({db, directiveId, taskId, question, options, deadlineAt})`. New `DaemonOptions.workerAuthToken` enables the route + builds the handler; `workerAskUserDefaultDeadlineSeconds` defaults 3600.
  - `apps/factoryd/src/main.ts` generates a per-startup 24-byte hex token via `crypto.randomBytes`, also exports it as `FACTORY5_WORKER_AUTH_TOKEN` env var (workers spawned by brain inherit it).
  - Tests: 14 ipc + 37 daemon (was 5 + 28 = 33; +18 net). Build/lint/format clean. 489 total workspace tests (was 471, +18).
  - **Carried forward to 8.3:** worker-side spawn must read `FACTORY5_WORKER_AUTH_TOKEN` from env and pass it as `Authorization: Bearer <token>` on every `/worker/ask-user` request. The MCP server (8.3) wraps this; `BRAIN_RPC_URL` derives from `loadDaemonEndpoint()`.

- [x] 8.3 — Worker-side MCP plumbing. Per ADR 0024 §1 the route is MCP. Scaffolded `packages/worker-mcp/` (new workspace package, +1 → 14 packages) with one tool: `ask_user(question, options?, deadlineSeconds?)`. Tool name claude exposes: `mcp__factory5-ask-user__ask_user`.
  - **`packages/worker-mcp/src/handler.ts`** — `askUserHandler({question, options, deadlineSeconds}, {env, fetchImpl?})`. Validates `BRAIN_RPC_URL` / `BRAIN_RPC_TOKEN` / `TASK_ID` / `DIRECTIVE_ID` are set (`AskUserEnvError` lists every missing var), POSTs to `<brainRpcUrl>/worker/ask-user` with `Authorization: Bearer <token>`, parses brain envelope, returns `{questionId, answer, timedOut, aborted}`. Network errors → `AskUserRpcError(0, 'NETWORK_ERROR', ...)`; non-2xx → `AskUserRpcError(status, code, message)` with upstream code if JSON envelope, `UPSTREAM_ERROR` otherwise. 11 tests (env validation × 2, happy paths × 4, URL/header/body asserts × 2, error paths × 3).
  - **`packages/worker-mcp/src/mcp-config.ts`** — `buildMcpConfig({scriptPath, brainRpcUrl, brainRpcToken, taskId, directiveId, nodeBin?})` returns the JSON object claude-cli expects. Exports `MCP_SERVER_NAME = 'factory5-ask-user'` and `ASK_USER_TOOL_NAME = 'mcp__factory5-ask-user__ask_user'` so 8.4 can add the latter to agent whitelists. 4 tests.
  - **`packages/worker-mcp/src/server.ts`** — MCP server entry; uses `@modelcontextprotocol/sdk` over stdio. Registers `ask_user` with JSON-schema input + heuristic-rich description (when to ask vs not). On `tools/call`, coerces args, runs handler, returns tool result with `isError: true` for timeouts / aborts (so the agent sees the failure mode and can fall back to a guess or raise a finding). No unit tests for the entry — exercised via 8.7 live validation; the protocol-handling layer is the SDK's responsibility.
  - **`packages/worker-mcp/src/index.ts`** — exports the helpers. `getServerScriptPath()` uses `import.meta.url` to locate `dist/server.js` portably across pnpm hoisting layouts.
  - **`packages/providers/src/types.ts` + `claude-cli.ts`** — added `ProviderRequest.mcpConfigPath`; `buildClaudeArgs` adds `--mcp-config <path>` when set. 2 new tests in `claude-cli.test.ts` (presence + empty-string skip).
  - **`packages/worker/src/run-worker.ts`** — added `WorkerAskUserConfig` interface (brainRpcUrl, brainRpcToken, directiveId) and `WorkerOptions.askUserConfig?`. In `runTooling`, when configured: write per-task mcp-config JSON to `os.tmpdir()/factory5-mcp-<taskId>.json` (out of the worktree so it never shows in `git status`), pass `mcpConfigPath` to `provider.stream()`, unlink in finally. Failed write → log warn, agent runs without `ask_user`. Adds `@factory5/worker-mcp` workspace dep.
  - **`packages/brain/src/pool.ts`** — `buildAskUserConfig(directiveId)` reads `FACTORY5_WORKER_AUTH_TOKEN` env (set by factoryd at 8.2 startup) and resolves the daemon endpoint via `loadDaemonEndpoint()`. Returns undefined when the env token is absent — covers tests and standalone scripts that drive the brain without the daemon shell. Only built for tool-using agents; read-only agents already have brain-checkpointed `escalateBlocked`.

- [x] 8.5 — `tasks_inflight` lifecycle + brain-restart recovery. **Migration 007** rebuilds `tasks_inflight` (SQLite-style — CHECK can't ALTER; new table → INSERT-SELECT → DROP → RENAME → reinstate indexes) widening `status` from 5 values to 7 (`+'waiting_for_human'`, `+'aborted'`) plus nullable `waiting_question_id` and `aborted_reason` columns.   - **`@factory5/core` `TASK_STATUSES`** widened to mirror.
  - **`@factory5/state` `tasks-inflight.ts`** gains `getById`, `markWaitingForHuman` (no-op unless current status = `'running'` — guards against downgrading aborted tasks), `markRunningAfterAnswer` (no-op unless `'waiting_for_human'` — race-safe vs. orphan recovery), `markAborted`, `findOrphanedHumanWaits`, `isTerminalStatus`. 15 tests covering happy paths + race-safe no-ops.
  - **`@factory5/state` `pending-questions.ts`** gains `detectOrphanedAnswer(db, qid)` returning `{taskId, taskStatus} | undefined` — used by channel collectors to surface the late-answer condition (treats missing task row as orphaned for forensic safety). 4 tests.
  - **`@factory5/brain` `ask-user.ts`** gains `onQuestionResolved?: (questionId) => void` callback — fires synchronously on new-question-create or rehydrated-open paths (skipped on rehydrated-already-answered).   - **§6.1 happy path.** `inject('POST /worker/ask-user')` runs; a polling `waitFor` observes the mid-flight `tasks_inflight.status = 'waiting_for_human'` with `waitingQuestionId` populated; answer is written to `pending_questions`; handler returns `{answer, timedOut:false, aborted:false}`; task flips back to `'running'` with `waitingQuestionId` cleared; `detectOrphanedAnswer` returns undefined (live consumer).
  - **§6.2 brain-restart mid-wait.** Seeds a `pending_questions` row + `tasks_inflight` row already in `'waiting_for_human'`, then runs `recoverFromHumanWaits(db, log)` directly. Asserts `recovered === 1`, status flipped to `'aborted'` with `abortedReason = 'brain_restart_during_human_wait'`, `finished_at` populated. Second call returns 0 (idempotent). A subsequent answer write is preserved for forensic value; `detectOrphanedAnswer` flags it as orphaned.
  - **§6.3 two-workers correlation.** Two parallel `inject()` calls on the same directive with distinct `taskId`s and distinct questions; `waitFor` blocks until both tasks are `'waiting_for_human'`; two writers answer each `waitingQuestionId`; both response bodies carry the correct answer and `questionId`; both tasks return to `'running'`; `pending_questions.task_id` partitioning verified.
  - **§6.4 late-answer no-op.** Task pre-aborted with `abortedReason='budget_exhausted'`; channel collector writes an answer to the linked question after-the-fact; row update preserved (answer + answeredAt set); `detectOrphanedAnswer` returns `{taskId, taskStatus:'aborted'}`; task row unchanged (terminal state, finishedAt, abortedReason preserved).
