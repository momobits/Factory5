Phase 5d landed I001 + I003 (prompt-tuning). The live runs exposed a new
HIGH-severity infrastructure bug — **I004**. When two sibling builders
merge their worktrees back to `main` concurrently on Windows, the
second merge is logged `worktree: merged and removed` but never lands
in main's reflog. The downstream CLI task then branches from a main
that's missing the lost sibling's files, re-creates a stub, and fails
merge-back with `Merging is not possible because you have unmerged
files`. Blocks `gate.build: true` end-to-end on any parallel-admitting
build.

This session fixes I004. Pure code + unit tests + one targeted live
rerun on `parallel-example` to confirm. No new ADR unless the
investigation surfaces an architectural gap (it shouldn't — this is a
concurrency bug, not a design choice).

Runs in parallel with the autoresume prompt (`startprompt-autoresume.txt`)
— I004 touches `packages/worker/`; autoresume touches `packages/daemon/`
+ `packages/state/` + `packages/cli/`. Disjoint file sets. Merge both
before running the close-out (`startprompt-phase5-closeout.txt`).

Step 1 — Read the existing implementation.

All in `packages/worker/src/worktree.ts`:

  a. `mergeAndRemove` (around line 215) — the function that loses
     commits. Note critically:
     - The merge runs on the **main project repo** (`simpleGit(projectPath)`),
       NOT inside the worktree (`wtGit` is only used to stage+commit
       the worker's output before merge).
     - It does `git checkout <baseBranch>; git merge --no-ff <branch>`
       on the shared project directory. Two concurrent callers both
       run this against the same `.git/` — they share `.git/index.lock`.
     - simple-git does not coordinate between multiple client
       instances against the same repo.

  b. `allocateWorktree` (around line 180) — branches from
     `<baseBranch>` at the moment the worktree is allocated. So if
     Worker-2's allocate runs AFTER Worker-1's merge completes,
     Worker-2 sees the updated main. This is correct and is NOT the
     race.

  c. `releaseWorktree` — the caller that picks success vs failure
     path, around line 260+. Confirm mergeAndRemove is only called on
     success.

Then read:

  d. `packages/brain/src/pool.ts` — specifically `executeTask` and the
     Promise.race draining loop. Confirm: the pool doesn't serialise
     the merge phase; each task calls `runWorker` which calls
     `releaseWorktree` independently of sibling tasks. That's the
     concurrency entry point we need to close.

  e. `packages/worker/src/run-worker.ts` — confirm `releaseWorktree`
     is invoked after the subprocess exits (success branch) and no
     earlier.

Write down what you see. The I004 issue file has two hypotheses
(Windows NTFS lock semantics; merge-in-main-vs-worktree). The code
points to hypothesis 1 (concurrent merges on a shared project repo);
hypothesis 2 is ruled out — the merge ALREADY happens in the main
checkout.

Step 2 — Pick a fix. Three candidate approaches:

  Approach A — project-level async mutex in `worktree.ts`.
    - Module-level `Map<string, Promise<void>>` keyed by normalized
      `projectPath`. `mergeAndRemove` chains onto the tail before doing
      the checkout + merge + remove sequence.
    - Low-risk, local change, preserves overall pool concurrency (only
      the merge phase serialises; the worker subprocesses still run
      concurrently).
    - Cross-platform — works on Linux too, where the race may or may
      not manifest.

  Approach B — queue-based serialisation in the pool.
    - `packages/brain/src/pool.ts` grows a merge-phase queue; the
      worker surfaces a "ready to merge" callback and the pool dispatches
      merges single-threaded.
    - More invasive. Puts concurrency policy in the pool, not the
      worker. Probably overkill.

  Approach C — use `git -C <path>` with retry-on-lock.
    - Wrap merge with a retry loop that catches index.lock errors and
      backs off. Does not eliminate the underlying race; papers over
      it. Rejected.

Ship **Approach A**. Small, local, testable.

Defense-in-depth (add regardless of approach):
  - After the merge completes, read `git rev-parse <baseBranch>` and
    compare against the pre-merge HEAD. If it didn't advance to the
    merge commit you just made, something silently failed — raise.
    Without this, the original bug re-appears if Approach A's mutex
    somehow doesn't cover every code path.

Step 3 — Implement.

  a. `packages/worker/src/worktree.ts`:
     - Add a module-level `Map<string, Promise<void>>` named
       `projectMergeQueues` (or similar). Key = normalized projectPath
       (use `path.resolve` so `.` and absolute forms collapse).
     - Wrap the body of `mergeAndRemove` in a function that chains onto
       the existing tail: read the current tail, create a new promise
       that awaits the tail then does the merge work, store that as the
       new tail. Clean up the entry when the promise settles to avoid
       leaking entries for dead projects.
     - Inside the critical section:
       1. Commit outstanding worker changes (already there).
       2. Record pre-merge main HEAD via `git rev-parse <baseBranch>`.
       3. Checkout main, merge `--no-ff`, as today.
       4. Record post-merge main HEAD. If equal to pre-merge HEAD
          (merge silently didn't advance), throw a clear error; the
          worktree stays preserved for inspection. Include both hashes
          in the error message.
       5. Remove worktree + branch as today.
     - Keep the existing "merge conflict → abort" error path;
       verification step is just an additional guard, not a replacement.

  b. `packages/worker/src/worktree.test.ts`:
     - Add a test that exercises two concurrent `mergeAndRemove` calls
       on the same project (pre-create two worktrees with disjoint
       files, kick off both merges in parallel, await both). Assert:
       both merges reach the reflog (use `git log main --oneline | wc
-l` or `git.log({ maxCount: 10 })` equivalent). Both files exist in
       `main`. Without the mutex this test should fail (reproduce the
       bug); with the mutex it should pass.
     - Add a test that exercises post-merge-HEAD verification: force a
       scenario where `git merge` returns success but HEAD doesn't
       advance (hard to construct naturally — a hand-crafted test git
       repo or a monkey-patched `git.raw` that returns success while
       leaving HEAD unchanged). If constructing the fault injection is
       messy, cover this with a smaller unit test against the
       verification helper itself rather than the full flow.
     - Keep the existing 9 worktree tests passing unchanged.

  c. No changes expected in `run-worker.ts` or `pool.ts` — they call
     `releaseWorktree` / `mergeAndRemove` and don't need to know about
     the mutex.

After each concrete edit: `pnpm build`, `pnpm --filter @factory5/worker test`
(fast iteration), then full `pnpm test` once the worker tests are green.
