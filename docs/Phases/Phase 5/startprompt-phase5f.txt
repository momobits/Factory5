Step 1 — Read the existing implementation.

All in `packages/assessor/src/runners/pytest.ts`:

  a. `pickPython(opts)` — the helper that chooses an interpreter.
     Today: prefers `<projectPath>/.venv/Scripts/python.exe` (Windows)
     or `<projectPath>/.venv/bin/python` (Unix) when present; parses
     `pyproject.toml`'s `requires-python` and prefers `py -3.11` /
     `python3.11` when the constraint calls for 3.11+; otherwise falls
     back to bare `python`. Note the return shape (likely a
     `{chosen, prefixArgs, version, reason}` object or similar).

  b. The assessor's install step. Today it runs `<chosen> [prefixArgs]
     -m pip install -e ".[dev]"` (falling back to `-e .`) from the
     project root. When `<chosen>` is the system interpreter, pip's
     default is to install into the current user's site
     (`%APPDATA%\Python\Python311\site-packages` on Windows,
     `~/.local/lib/python3.11/site-packages` on Linux) and register a
     `.pth` that puts the project's `src/` on every subsequent
     invocation's `sys.path`. That's I006.

  c. The pytest invocation step. Same `<chosen> [prefixArgs] -m pytest
     ...` — uses whichever interpreter the install step used, so once
     we pivot to a venv for install, pytest must pivot with it.

  d. The import-check step (`checkPythonImports` if that's the name)
     — also uses the picked interpreter. Must pivot too.

Skim `packages/assessor/src/runners/pytest.test.ts` so you know the
existing tests' shape before adding new ones.

Step 2 — Pick a fix tier. I006's issue file enumerates three:

  Tier 1 — per-project isolated venv in `.factory/assessor-env/`.
  Tier 2 — `pip install --target <tmpdir>` + PYTHONPATH injection.
  Tier 3 — pre-uninstall the project's distribution name before install.

**Ship tier 1.** It is the smallest, cleanest change that fully closes
the issue, aligns with ADR 0017's direction (assessor provisions its
own env), and gracefully reuses `.factory/`'s gitignored home.
Tier 2's PYTHONPATH routing is fiddly around console-scripts and
binary wheels; tier 3 is a loud mutation of the user's global env that
only covers the same-name-repeated case.

Respect ADR 0017's precedence: **if the project already has its own
`<projectPath>/.venv/`, prefer it** (user-controlled env). Only
fall through to `.factory/assessor-env/` when no project-level venv
exists. This keeps a developer's deliberate venv setup authoritative.

Step 3 — Implement.

  a. `packages/assessor/src/runners/pytest.ts`:

     - New helper `ensureAssessorVenv({ projectPath, basePython })`:
       1. If `pickProjectVenv(projectPath)` returns a valid venv
          interpreter (i.e. the existing `.venv/` precedence), return
          that — don't create an assessor-managed env.
       2. Otherwise compute
          `envPath = join(projectPath, '.factory', 'assessor-env')`.
          Platform-aware interpreter path:
          - Windows: `join(envPath, 'Scripts', 'python.exe')`
          - Unix:    `join(envPath, 'bin', 'python')`
       3. If the interpreter file doesn't exist, create the venv:
          spawn `basePython.chosen [basePython.prefixArgs] -m venv
          <envPath>` with `{ shell: false }` and explicit args.
          Surface stderr on failure; don't swallow.
       4. If creation succeeded, log at `info` with `envPath` +
          `basePython.version` so ops can trace which interpreter
          seeded the venv.
       5. Return a `PickedPython`-shaped object pointing at the venv's
          interpreter with **empty `prefixArgs`** (a venv's python is
          invoked directly, not via `py -3.11`).
       6. Export the helper so the test file can unit-test it.

     - Thread the helper into the assessor's main flow:
         basePicked = pickPython(opts)
         picked     = await ensureAssessorVenv({ projectPath, basePython: basePicked })
         install(picked, ...)    // unchanged
         pytest(picked, ...)     // unchanged
         checkImports(picked, ...) // unchanged
       Every downstream step that today uses `basePicked` must use
       `picked`.

     - `AssessResult.provisioning` gets a new optional field:
       `venvSource: 'project' | 'factory-managed' | 'system'` — 'project'
       when `.venv/` already existed, 'factory-managed' when we
       created `.factory/assessor-env/`, 'system' only if the fallback
       somehow lands on a non-venv interpreter (should not happen once
       tier 1 lands; log a warn if it does). Useful for debugging.

     - Cache by reuse: the "is the interpreter file present?" check at
       step 3 IS the cache. Incremental assesses on the same project
       get a hot venv — no recreation, no re-install of base tooling.
       Package install still runs every assess (cheap in incremental
       mode — pip detects no changes and exits fast).

  b. `packages/assessor/src/runners/pytest.test.ts`:

     - `ensureAssessorVenv returns project .venv when present` — seed a
       fake `<tmp>/.venv/Scripts/python.exe` (or `bin/python`), assert
       the helper returns that without creating `.factory/assessor-env/`.
     - `ensureAssessorVenv creates .factory/assessor-env/ when no
       project venv` — no fake `.venv/`, stub out the subprocess spawn
       to simulate `python -m venv`, assert creation and correct
       interpreter path returned. Don't actually shell out to python
       in unit tests — that's a live integration.
     - `ensureAssessorVenv reuses an existing assessor venv` — seed a
       pre-existing `.factory/assessor-env/.../python` file, call
       twice, assert the spawn stub is called zero times.
     - Platform handling: one parametrised case for Windows path
       (`Scripts/python.exe`) and one for POSIX (`bin/python`). Use
       `process.platform` mocking via your existing pattern (grep the
       test file for how `pickPython` already handles this — mirror
       that).

     - End-to-end assess test (if one already exists) — extend it to
       assert `provisioning.venvSource` is `'factory-managed'` on a
       project without `.venv/`, and that the install went to the
       factory venv (check for `<projectPath>/.factory/assessor-env/`
       existing on disk).

     - Aim for +3-5 tests. Workspace total should go from 247 to
       ~250-252.

  c. `docs/decisions/0017-assessor-project-env-provisioning.md`:

     - Do **not** write a new ADR. Add a short "Implementation notes"
       section (or extend an existing one) to 0017 describing:
       - Precedence: project `.venv/` → `.factory/assessor-env/` →
         system (deprecated fallback).
       - Why `.factory/assessor-env/` (gitignored, per-project,
         reused across incremental assesses).
       - Pointer to I006 for the incident history.

  d. No brain / worker / wiki / CLI / prompts changes. If you find
     yourself tempted to touch those, stop — you're out of scope.
