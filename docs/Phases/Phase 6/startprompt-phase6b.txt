---

Why now — the forcing function.

Factory has only ever been driven by the CLI. Discord channel
exists but is underused; no non-CLI build triggers. Phase 5
produced a credible autonomous loop and green gates — the next
leverage is exposing the loop to external callers. GitHub is the
most natural first external trigger because: (a) the projects
factory builds end up in GitHub repos, (b) GitHub's issue / PR
surface maps naturally to directives, (c) most tooling already
assumes GitHub event plumbing.

6b doesn't block 6a / 6c, but it's the biggest sub-phase —
budget accordingly.

Step 1 — Understand the existing channel plumbing.

Channels today live in `@factory5/channels`. Core surface:

  - `types.ts` — `Channel` interface. Methods (paraphrased; read
    the real file):
    - `id: string` — unique channel identifier.
    - `start(deps)` / `stop()` — lifecycle.
    - Some form of message-in handler that produces
      `directive:new` events.
    - Some form of message-out sender (posting findings,
      terminal status back to the channel's surface).
  - `registry.ts` — registers channels, dispatches events.
  - `cli-rpc.ts` — the CLI's transport channel. Simplest
    reference for "how a channel receives commands."
  - `discord.ts` — `DiscordChannel` via `discord.js`. Full
    implementation; ~300+ lines; covers auth (bot token), event
    subscription, message parsing, reply threading.

Also read:

  - `packages/daemon/src/` — channel registration in the daemon
    lifecycle. Grep for `DiscordChannel` / `registerChannel` to
    find the wiring point.
  - `packages/state/src/queries/outbound.ts` — outbound-message
    queue. Channels post back through this queue so the daemon
    can retry on failure.

Confirm your understanding before picking a design: a channel is a
bidirectional edge between factory's directive pipeline and an
external surface. It consumes events (incoming messages) and
produces events (outgoing findings/status). The channel does NOT
drive the pipeline — it emits `directive:new` and lets the brain
take over.

Step 2 — Design decisions.

Three coupled choices. Pick each before writing code.

**Choice 1 — Auth model.**

_Option A: Personal Access Token (PAT)._ User generates a PAT with
`repo` + `issues` scopes, pastes into config. Octokit uses it for
all API calls. Simple; single-user; no webhook receiver required.

_Option B: GitHub App._ Factory is registered as a GitHub App;
users install it on their orgs / repos. Supports webhook receive,
PR comments, multi-repo. More complex; requires factory to host a
public webhook endpoint.

**Recommendation: PAT for v1.** Ship the channel on polling +
PAT; defer GitHub App + webhook to v2 once v1 is proven.

**Choice 2 — Transport.**

_Option A: Polling._ Factory polls the GitHub API every N seconds
for new issues / comments matching trigger rules. Simple; no
public endpoint; rate-limit aware. Latency: 15-60s depending on
poll interval.

_Option B: Webhook receive._ Factory exposes an HTTP endpoint;
GitHub POSTs events there on-change. Low latency; requires public
endpoint (ngrok tunnel for local dev, Cloudflare Tunnel / fly.io
for production).

**Recommendation: Polling for v1.** The latency is fine for build
jobs (which take minutes anyway). Webhook lands in a follow-on
session once the polling version is proven and the user commits
to a hosting story.

**Choice 3 — Trigger grammar.**

_Option A: Label-driven._ Issues with label `factory/build` or
`factory/triage` produce directives. Clean; user controls with
native GitHub UI.

_Option B: Comment command._ Comments on issues / PRs starting
with `/factory <intent>` (e.g. `/factory build`,
`/factory triage`) produce directives. Flexible; supports PR
comment scope.

_Option C: Both._

**Recommendation: Both, label first.** Label-driven is easier to
implement (one API call per poll returns labelled issues); comment
command adds PR scope and is the natural next step. Implement
label first, then comment command as a second commit in the same
session if time permits.

Step 4 — Implement v1.

4.1 — Dependency + config.

  - Add `@octokit/rest` to `packages/channels/package.json`. (It's
    the official GitHub API client; mature, well-typed.) Run
    `pnpm install -w --filter @factory5/channels @octokit/rest`.
  - Extend the factory config shape — probably in
    `packages/core/src/schemas.ts` or wherever
    `DiscordChannelConfig` lives — with a
    `GithubChannelConfig` shape: `{ pat: string, repos: string[],
    pollIntervalMs?: number, triggerLabel?: string }`.
  - Config discovery: mirror Discord's convention (env var? secret
    file under `~/.factory5/secrets/`? Check Discord's pattern
    first).

4.2 — Channel implementation.

  - `packages/channels/src/github-channel.ts` —
    `class GithubChannel implements Channel`.
    - Constructor takes config + Octokit instance (injected for
      test seam).
    - `start()`: start the poll loop (setInterval-based, cancel on
      `stop()`). Each poll: for each configured repo, fetch
      open issues with `triggerLabel` since last-poll timestamp.
    - For each matching issue: emit a `directive:new` event with
      `source = 'github'`, `sourceRef = 'github:${owner}/${repo}#${issueNumber}'`, intent parsed from label or
      comment body. Remove the label after processing (so we
      don't re-trigger next poll — this is the "poll-as-idempotency"
      hack; document it).
    - Outbound: listen for outbound-message events scoped to
      `github:*` refs, post them as issue comments via
      `octokit.issues.createComment`.
  - Mirror `discord.ts`'s shape closely — same lifecycle, same
    error-handling discipline (Pino logger, retry on transient
    API failures, cancel on shutdown signal).

4.3 — Registration in the daemon.

  - Grep for where `DiscordChannel` is instantiated in
    `apps/factoryd/` or `packages/daemon/`. Add a conditional
    GithubChannel instantiation there, gated on config presence.
  - `factory doctor` probe: add a `probeGithub` (if GithubChannel
    config is present, verify the PAT + repo access). Skip via
    `--skip-github` flag.

4.4 — Tests.

  - `packages/channels/src/github-channel.test.ts` —
    deps-injection seam for Octokit. Test cases:
    - Poll finds an issue with trigger label → `directive:new`
      emitted with correct source/ref/intent.
    - Poll finds nothing new → no event.
    - Outbound message posted to the correct issue.
    - Transient 502 → retry; persistent 401 → surface error,
      don't retry.
    - Aim for +5-7 tests.
  - No real Octokit calls in unit tests. Integration test (optional)
    under `packages/channels/test/` using a recorded-response
    shim if the existing channels package has a pattern for that.

Full workspace test count should move from 255 to ~260-265.

Step 5 — Live validation.

Budget: $4-8.

Pre-requisites (do BEFORE the live run):

  1. User creates a GitHub PAT with `repo` + `issues` scopes.
     Paste into the configured secret location.
  2. User creates a **tiny test repo** (e.g.
     `momobits/factory-6b-test`) with one issue templated as:
     - Title: "Build a weather CLI (factory test)"
     - Body: brief description mirroring the `example` spec.
     - Label: `factory/build` (create the label first).
  3. `factory doctor --skip-discord` green.
  4. `factoryd` not running (fresh start).

RUN:

  factoryd start  # config now includes GithubChannel

Expected signals (from `factoryd` logs):

  - `channels.github: started` on startup.
  - Within `pollIntervalMs`: `channels.github: poll found issue
    owner/repo#N` for the test issue.
  - `directive:new` emitted with
    `source: 'github', sourceRef: 'github:owner/repo#N'`.
  - Brain picks up the directive; triage → architect → planner →
    pool → assess → complete. Just like a CLI-triggered build.
  - `channels.github: posted comment` for each outbound (at
    minimum: findings-raised comments + terminal status comment).
  - On the GitHub issue: a comment from factory's PAT-bearing
    account containing the terminal status / gate line.
