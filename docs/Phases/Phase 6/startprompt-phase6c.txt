**Path A — Verifier gets filesystem access (upgrade to tool-using).**

Structural fix. The verifier becomes authoritative: if it raises a
finding, it had to actually see the problem.

Concrete changes:

  - `packages/worker/src/run-worker.ts` — `isToolUsingAgent(role)`
    currently returns true for `scaffolder | builder | fixer`. Add
    `verifier`. That flips the verifier onto the `runTooling`
    code path: worktree allocation + `provider.stream()` with
    `allowedTools`.
  - Allowed tools: `['Read', 'Glob', 'Grep']` — read-only set. The
    verifier should NOT get Write/Edit/Bash; it's a verifier, not a
    builder. If you think Bash is needed for `git log` / test
    counting, whitelist `Bash` but add a guard note in the prompt.
  - Verifier already runs inside a worktree post-pool, so it sees
    the post-merge state. Cleanup follows the existing
    `runTooling` → `cleanupWorktree` path.
  - `prompts/agents/verifier.md` — flesh out from stub. Core rule:
    "Do not claim a file is absent unless you have attempted to
    read it and got a `not found` error. Every assertion about the
    filesystem MUST be grounded in a tool call in this turn." Add
    a concrete worked example.
  - Tests: +2-3 worker tests exercising the verifier-tool-using
    path (worktree allocated, read-only allowlist enforced).

Consequences to weigh:

  - Cost: verifier moves from single-shot call (~15s) to streaming
    with tool turns (~60-120s). Spend per run goes up maybe $0.30.
  - Worktree allocation cost: +1 worktree per directive. The merge
    race was a pain point pre-I004; the mutex fixed it but one
    more worktree doubles the pressure on that code path.
  - This says "verifier's findings are authoritative." That's a
    stance — the assessor is still ground truth for gate booleans,
    but the verifier's findings now carry weight instead of being
    recommended-ignore.

**Path B — Verifier findings become advisory (no gate impact).**

Minimum-change fix. Formally encode what was already implicit: the
verifier is LLM-based second opinion, not authority. Its findings
surface for operator attention but never block the loop.

Concrete changes:

  - `packages/core/src/schemas.ts` — add `advisory: boolean` to the
    `Finding` shape (default `false`).
  - `packages/wiki/src/findings.ts` (or wherever `addFinding` lives)
    — accept `advisory` in the input; persist it.
  - `packages/worker/src/run-worker.ts` — `persistFindings()` is
    called by both `runReadOnly` and `runTooling`. Thread
    `advisory = true` when `task.agent === 'verifier'`. Consider
    whether other read-only agents (architect, planner, triage)
    should also be advisory — probably yes; they're all LLM-based
    non-authoritative signals. That's a scope decision.
  - `packages/brain/src/loop.ts` — wherever `openFindings` feeds
    into terminal-status / escalation logic, filter out advisory
    findings. Keep them in the `listFindings` API so the operator
    still sees them.
  - `prompts/agents/verifier.md` — flesh out from stub. Core rule:
    "Your findings are advisory, never blocking. Surface risks the
    operator should know about; do not claim authority over the
    build's correctness — that is the assessor's job." Worked
    example showing the shape of an advisory finding vs. a
    hallucination.
  - `factory findings list` — default behaviour decision: show
    advisory inline (distinguished visually) or hide behind
    `--include-advisory`? Probably show inline, with a marker
    like `[advisory]` in the listing.
  - Tests: +3-4 tests: (a) verifier-raised findings carry
    `advisory: true`; (b) builder-raised findings do not;
    (c) `openFindings` blocker logic excludes advisory; (d) CLI
    output differentiates advisory vs. authoritative.

Consequences to weigh:

  - The verifier can still hallucinate — it just doesn't matter
    anymore. Cosmetic fix; some operators may prefer a verifier
    that actually knows what it's claiming.
  - Cleaner surface area. No worktree churn for the verifier; no
    tool budget spent on read-only filesystem scans.
  - Signals a design choice: factory's verification authority is
    ground-truth (assessor) only; LLM signals are aids, not
    judgments.

---

**Recommendation — Path B.** It's the smaller, more honest change.
Path A pushes the verifier toward authority; but the assessor is
already the authority for gate booleans (ADR 0017), and doubling up
the authority path is wasted effort when the assessor is the
authoritative one.

Path B makes the split explicit: assessor = authority, verifier (and
architect, triage, planner when they raise findings) = advisory.
That's a cleaner mental model, smaller code change, and doesn't
buy into the "LLM can verify filesystems" assumption that F001
already falsified.

But this is a design call — if you disagree, argue for A and ship
that. Don't split the difference.
