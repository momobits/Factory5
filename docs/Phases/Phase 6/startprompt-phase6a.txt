Step 1 — Understand the existing state.

Three datastores interact today:

  a. **Per-project findings files** —
     `<workspace>/<project>/.factory/findings.json`. JSON,
     atomic-write via `writeFileAtomic`, shape
     `{ nextSequence: N, findings: Finding[] }`. Authoritative for
     "what findings exist on THIS project." Written from
     `@factory5/wiki` (`addFinding`, `updateFindingStatus`).

  b. **Factory-home state db** —
     `~/.factory5/state.sqlite`, managed by `@factory5/state`.
     Tables: `directives`, `tasks_inflight`, `sessions`, `events`,
     `model_usage`, `pending_questions`, `projects`, `learnings`,
     `outbound_messages`. Does NOT currently index findings —
     the brain has never needed to cross-reference them.

  c. **Factory-home wiki cache** — doesn't exist. Each project's
     wiki is read fresh from disk when the brain needs it.

The pattern: sqlite holds runtime orchestration state; JSON files
hold per-project ground-truth artefacts. The registry should
respect that split — don't dump the entire `findings.json` into
sqlite; instead index the fields the registry needs to query
across (severity, status, source, project, finding_id,
created_at) and keep the JSON files authoritative for description
bodies.

---

Step 2 — Design decision.

Two paths. Pick one before writing code.

**Path A — Integrated registry table in `state.sqlite`.**

Add a `findings_registry` table to the existing state db via a
new `003-findings-registry.ts` migration. Columns:

  - `project_path TEXT NOT NULL` (the `<workspace>/<project>` dir)
  - `finding_id TEXT NOT NULL` (e.g. `F001`)
  - `source TEXT NOT NULL` (agent role that raised it)
  - `target TEXT NOT NULL`
  - `severity TEXT NOT NULL CHECK (severity IN ('LOW','MEDIUM','HIGH','CRITICAL'))`
  - `status TEXT NOT NULL CHECK (status IN ('OPEN','RESOLVED','WONTFIX','ADVISORY'))`
  - `description TEXT NOT NULL` (denormalised — acceptable cost
    for the query convenience; schema migration can drop it later
    if it grows too big)
  - `created_at INTEGER NOT NULL` (epoch ms)
  - `updated_at INTEGER NOT NULL`
  - PRIMARY KEY (`project_path`, `finding_id`)
  - index on (`severity`, `status`)

Sync mechanism: **write-through**. Patch `wiki/src/findings.ts`'s
`addFinding` + `updateFindingStatus` so each finding-lifecycle
mutation also upserts into the registry. This requires
`@factory5/wiki` to depend on `@factory5/state` — check the
workspace DAG for cycles; today wiki → core, state → core, no
cycle added by wiki → state.

Read path: `factory findings list` opens state.sqlite directly
(read-only `better-sqlite3` handle) — no daemon round-trip.

Pros: queries are fast, cross-project by default, one source of
truth for "all OPEN HIGH findings." Write cost is one extra
sqlite upsert per finding — cheap.

Cons: registry can drift if a finding is added to findings.json
outside the `addFinding` API (shouldn't happen, but possible).
Recovery path: `factory findings resync <projectPath>` rescans a
specific project and rewrites its registry rows.

**Path B — Reconstruct on demand from filesystem scan.**

No sqlite table. `factory findings list` walks a configured list
of `<workspace>` dirs, reads each `<project>/.factory/findings.json`,
concatenates + filters in-memory.

Pros: zero sync risk — the JSON files are the single source of
truth. No migration, no write-through.

Cons: slow as project count grows; requires maintaining a
workspaces-list config
(`~/.factory5/workspaces.json`? Or scan under `~/`?). The
"walk disk every query" model scales badly; will need to become
Path A eventually anyway.

---

**Recommendation — Path A.** It's more code up front but the
registry is a write-through index, not a cache — reads never miss.
The cost of getting it wrong (stale rows) is a `resync` command,
which is cheap to implement and user-visible as an escape hatch.
Path B defers the real work and imposes a UX penalty (slow list
commands) that compounds as Phase 6+ adds more projects.

But argue the other way if you see a reason. Don't split the
difference with "sqlite cache that rebuilds on miss" — that's the
worst of both.

Title (Path A): `0018-cross-project-findings-registry`. Context:
Phase 5 produced multiple live workspaces, no cross-project view,

Step 4 — Implement Path A.

4.1 — Migration.
  - `packages/state/src/migrations/003-findings-registry.ts` with
    the table + indexes per Step 2.
  - Wire into `migrations/index.ts`. Write a migration test mirroring
    `002-directive-blocked-reason.ts`'s shape (see
    `packages/state/src/state.test.ts` for the existing migration
    test pattern).

4.2 — Queries.
  - `packages/state/src/queries/findings-registry.ts` —
    `upsertFinding(db, row)`, `listFindings(db, filter)`,
    `countBySeverity(db, filter)`,
    `deleteByProject(db, projectPath)` (used by `resync`).
  - Mirror the shape of `queries/directives.ts`. Include Zod
    validation on the filter inputs.
  - Unit tests parallel to `queries/directives.test.ts`. Aim for
    +6-8 state tests.

4.3 — Wiki integration (the write-through).
  - `packages/wiki/src/findings.ts` — `addFinding` and
    `updateFindingStatus` each take an optional
    `registry?: RegistryHandle` parameter. When provided, upsert
    the finding post-write. Keep the parameter optional so
    existing callers (tests, inline utilities) don't break.
  - `packages/wiki/package.json` — add `@factory5/state` to
    deps.
  - Test: stub a registry, call `addFinding(projectPath, {...},
    stub)`, assert upsert was called with the expected row shape.
  - Tests: +2 wiki tests.

4.4 — Brain / daemon wiring.
  - `packages/brain/src/loop.ts` + wherever `addFinding` is
    called in the brain. Thread the state-db handle (already
    available on the brain's state object) down to the wiki
    calls. Minimally invasive.
  - Consider whether `deleteByProject` fires when a project is
    deleted — probably out of scope (`factory project forget`
    doesn't exist); skip.

4.5 — CLI command.
  - `packages/cli/src/commands/findings.ts` — `factory findings`
    Commander sub-tree:
    - `factory findings list [--severity HIGH] [--status OPEN]
      [--project <glob>] [--source <agent>] [--format text|json]`.
      Default: `--status OPEN`, sort by severity desc, created_at
      asc.
    - `factory findings show <project-glob> <finding-id>` —
      reads the JSON file for the full description body (registry
      only has the short form).
    - `factory findings resync <project-path>` — read the JSON,
      rewrite registry rows for that project.
  - Wire into the CLI entrypoint (`packages/cli/src/cli.ts`).
  - Tests: +3-4 CLI tests covering filter combinations and
    resync. Use a tmp state db and tmp project dirs.

4.6 — Backfill utility.
  - One-time `factory findings resync-all <workspaces...>` OR
    implicit first-run backfill. Pick the explicit command —
    implicit first-run is magic; explicit command is visible.
  - Documentation: add a short "Migrating existing projects" note
    to the new ADR.

DO NOT in this session:

  - Start 6b or 6c. Phase 6 sub-phases are independent; pick one.
  - Dump the entire `findings.json` body into sqlite. Registry is
    an index, not a replacement.
  - Add a daemon-hosted "registry server" — the CLI reads sqlite
    directly, same shape as `factory directive list`.

    mechanism for drift that we expect not to happen.

    is file+sqlite hybrid by ADR 0003; stay in that discipline.
