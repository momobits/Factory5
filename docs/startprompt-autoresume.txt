Step 2 — Pick a path. Two are acceptable, both land in this session:

  Path A — `factory directive mark-blocked <id> [--reason <text>]` CLI.
    - New command (new file `packages/cli/src/commands/directive.ts` or
      extended existing structure).
    - Calls a new `directivesQ.markBlocked(db, id, reason?)` that sets
      status to `blocked` with an optional recorded reason.
    - Argv parsing validates the ID exists + is currently `running`.
      Refuses to mark non-running directives (safety).
    - Operator runs manually when they know a directive is dead.

  Path B — factoryd-start reconciliation.
    - On factoryd startup (before serving), scan directives where
      status=`running` AND (claimedBy is set AND looks like a dead PID) OR
      (no heartbeat in N minutes — pick a threshold that won't false-flag
      a healthy long run, e.g. 10 min since last `model_usage` row).
    - Mark them blocked, with reason like "reconciled at startup — prior
      process gone".
    - Log at warn level.
    - Unit-test the reconciliation function against a seeded DB.

Recommended: ship both. Path A is 30 lines + a test and gives the operator
an immediate hand-crank. Path B automates the common case but needs the
heuristic right. Ship A first, then B as a second commit.

If you can only do one in this session: do A. Stuck directives happen
rarely enough that a CLI is enough.

Step 3 — Implement Path A.

  a. `packages/state/src/queries/directives.ts`:
     - New `markBlocked(db, id, reason?)`. Looks up the directive;
       returns an error if not found or already terminal. Updates status
       to `blocked`. If a `blocked_reason` / `metadata` column exists,
       record the reason; if not, add a migration OR log it out via the
       CLI. Prefer adding a migration — future log inspection benefits.

  b. `packages/state/src/migrations/` — add a new migration (next id)
     that adds a nullable `blocked_reason TEXT` to the directives table
     IF it's not already there. Read existing migrations for shape.

  c. `packages/cli/src/commands/` — new command:
         factory directive mark-blocked <id> [--reason <text>]
     Wire it from `packages/cli/src/index.ts` alongside existing commands.

  d. Tests:
     - `packages/state/src/queries/directives.test.ts` (or create it) —
       test markBlocked: happy, not-found error, already-blocked error.
     - CLI test if the CLI package already has them; otherwise skip for
       Path A.

After each concrete edit: `pnpm build`, `pnpm --filter @factory5/state test`,
`pnpm --filter @factory5/cli test` (if present), `pnpm lint`,
`pnpm format:check`.

Step 4 — Implement Path B (optional in this session).

  a. `packages/daemon/src/index.ts` (or wherever startup sequencing
     lives):
     - After `runMigrations`, before starting channels, call a new
       `reconcileOrphanedDirectives(db, logger)` from
       `packages/state/src/queries/directives.ts`.

  b. `reconcileOrphanedDirectives`:
     - Find directives where status=`running` AND (claimedBy NULL OR the
       claiming PID is gone AND no `model_usage` row in last 10 min).
     - For each: call `markBlocked(db, id, 'reconciled at daemon start')`.
     - Return the count reconciled so the daemon can log it.
     - Cross-platform PID liveness check: use `process.kill(pid, 0)`
       inside a try/catch — throws if the PID is dead. This is the
       standard node idiom.

  c. Unit tests: seed a DB with (a) a live directive, (b) a fake-PID
     directive, (c) a terminal directive, then call reconcile. Assert
     only (b) flips to blocked.

  d. Don't integration-test factoryd startup for this — the unit test
     on the reconcile function is enough; the wiring is trivial.

After: `pnpm build`, `pnpm test` (full workspace — state + daemon), lint,
format:check.
